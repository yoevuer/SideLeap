## 1. 动作体系实现方式

### 1.1a [P1] actionText / actionIcon 映射数据化

**观察事实**：
- `ktx/GlobalActions.kt` 的 `actionText()`（第 55-110 行、第 112-168 行）和 `actionIcon()`（第 170-231 行）包含三个完整 `when` 分支，每个分支枚举相同的 30+ 动作 ID
- `actionText` 存在两个版本（`Context` 扩展函数和 `@Composable` 函数），逻辑完全相同仅获取资源方式不同（`getString` vs `stringResource`）
- 添加新动作需同步修改三个函数

**影响**：
维护负担：添加一个新动作需要同时修改 3 个位置，遗漏将导致 UI 无文本或图标。

**建议处理**：
将 actionText / actionIcon 映射替换为 `Map<String, Pair<Int, Any?>>` 数据结构，两个 `actionText` 版本共享同一个映射源。

**风险等级**：低

**是否需要用户确认**：是（涉及函数签名和数据结构变更）

**是否改变用户可见行为**：否（映射关系不变，仅实现方式从 `when` 分支改为 Map 查找）

**优先级**：P1

**推荐验证方式**：assembleDebug + 遍历所有 30+ 动作验证文本和图标显示

**是否适合自动化执行**：适合自动化

### 1.1b [P1] onAction ActionHandler 重构

**观察事实**：
- `SideGestureServiceProxy.kt`（code_index 核心链路索引"动作体系链路"）的 `onAction()` 方法包含约 250 行的 `when` 分支（对应 code_index 第 149-160 行所述"动作体系链路"的入口与执行逻辑）
- 每个分支包含不同的执行逻辑：系统 API 调用、协程、Toast、权限检查、数据反序列化、Intent 构建等
- 添加新动作需同步修改 4 个位置（`constant/GlobalActions.kt` 的 ID + `SideGestureServiceProxy.kt` 的 `onAction` + `ktx/GlobalActions.kt` 的 `actionText` + `actionIcon`）
- `onAction` 分支中的执行逻辑差异大，无法用纯数据映射替代

**影响**：
维护负担高，遗漏同步修改可导致动作 ID 有效但执行时无响应。

**建议处理**：
引入 `ActionHandler` 接口，将每个动作的执行逻辑封装到独立的实现类中，通过注册表消除 `when` 分支。

**风险等级**：中（涉及多文件重构和接口设计决策）

**是否需要用户确认**：是（涉及包结构调整，新建 ActionHandler 接口与实现类）

**是否改变用户可见行为**：否（仅内部重构，不改变动作执行结果）

**优先级**：P1

**推荐验证方式**：assembleDebug + 遍历所有 30+ 动作实机验证

**是否适合自动化执行**：需人工确认后自动化

### 1.2 [P1] 遗留 Legacy 动作 ID 的完整分支仍存活

**观察事实**：
- `constant/GlobalActions.kt` 第 79-93 行注释声明"以下常量不再出现在动作选择列表中，仅为旧配置兼容"
- 但 `ktx/GlobalActions.kt` 的 `actionText()` 和 `actionIcon()` 仍为这些 Legacy ID（MENU、SEARCH_IN_APP、KILL_APP 等约 20 个）保留完整分支
- 这些分支在 UI 中不可选，仅在旧版本已保存的配置被执行时命中

**影响**：
约 60 行死代码，误导新开发者以为这些功能仍活跃。

**建议处理**：
确认从 DataStore 获取的旧配置是否仍可能包含这些 ID。如果旧配置升级路径已完成，可删除对应分支。

**风险等级**：低

**是否需要用户确认**：是（需确认旧配置迁移逻辑是否确保这些 ID 已被清理或替换）

**是否改变用户可见行为**：否（仅删除不可达分支）

**优先级**：P1

**推荐验证方式**：grep 检索旧配置路径 + assembleDebug + 实机升级旧版配置验证

**是否适合自动化执行**：需人工确认后自动化

### 1.3 [P2] Action 数据以原始 String 传递，无编译期类型安全

**观察事实**：
- `entity/GestureActions.kt` 中 `Action.data` 定义为 `String`
- `SideGestureServiceProxy.kt` 的反序列化使用 `JsonHelper.decodeFromString<T>()` 配合 `try-catch`
- 每个动作期望的数据结构不同，但无编译期约束

**影响**：
运行时异常风险，数据损坏后难以调试。

**建议处理**：
将 `Action.data` 改为密封类或带泛型的 `ActionResult` 类型，在编译期约束各动作所需数据。

**风险等级**：低

**是否需要用户确认**：是（涉及数据模型变更，需确认序列化兼容性）

**是否改变用户可见行为**：否（不影响持久化后的语义）

**优先级**：P2

**推荐验证方式**：assembleDebug + 实机验证动作执行 + 从旧版本升级

**是否适合自动化执行**：需人工确认后自动化

## 2. 应用列表与应用状态查询

### 2.1 [P1] 包管理器查询存在重复循环

**观察事实**：
- `utils/AppInfoUtils.kt`（code_index 核心链路索引"应用列表链路"的执行逻辑层）中 `queryFrozenApplications` 和 `inspectDisabledAppsByPackageManager` 各自独立调用 `pm.getInstalledApplications(0)` 并遍历
- `getInstalledApplications` 是昂贵的系统调用，两个方法的循环体结构几乎相同

**影响**：
当两个方法被连续调用时，包管理器迭代量翻倍，在应用数量多的设备上可造成数十毫秒的额外延迟。

**建议处理**：
将 `pm.getInstalledApplications(0)` 结果缓存或合并为一次遍历。

**风险等级**：低

**是否需要用户确认**：否

**是否改变用户可见行为**：否

**优先级**：P1

**推荐验证方式**：grep 检索调用链 + assembleDebug

**是否适合自动化执行**：适合自动化

### 2.2 [P1] findLauncherActivity 使用 Thread.sleep 阻塞

**观察事实**：
- `utils/AppInfoUtils.kt` 第 225-254 行 `findLauncherActivity` 使用 `Thread.sleep(retryDelayMs)` 实现重试
- 最大重试 5 次 × 200ms = 最多阻塞 1 秒
- 调用方 `SideGestureServiceProxy.kt` 已使用 `Dispatchers.IO`

**影响**：
即使运行在 IO 调度器上，阻塞仍在浪费线程资源；当应用被冻结且找不到启动 Activity 时，1 秒的重试周期完全浪费。

**建议处理**：
替换为 `delay()` 协程挂起，避免线程阻塞。

**风险等级**：低

**是否需要用户确认**：否

**是否改变用户可见行为**：否

**优先级**：P1

**推荐验证方式**：grep 检索调用链 + assembleDebug

**是否适合自动化执行**：适合自动化

### 2.3 [P2] Pinyin 缓存无生命周期管理

**观察事实**：
- `utils/AppSearchUtils.kt` 第 11-15 行 `pinyinIndexCache` 是顶层 `LinkedHashMap`，最大 1024 条
- 在配置变更或应用生命周期回调中不清除

**影响**：
内存占用随应用浏览量线性增长。1024 条 × 3 字符串 = 约数万字符，内存影响有限但存在泄漏通道。

**建议处理**：
添加 LRU 淘汰或生命周期感知清理。

**风险等级**：低

**是否需要用户确认**：否

**是否改变用户可见行为**：否

**优先级**：P2

**推荐验证方式**：grep 检索引用 + assembleDebug

**是否适合自动化执行**：适合自动化

## 3. Compose UI 与状态派生

### 3.1 [P1] 无障碍服务覆层中同时订阅 5 个 DataStore StateFlow

**观察事实**：
- `SideGestureService.kt` 第 221-270 行在 Compose 覆层中创建 5 个独立的 `collectAsStateWithLifecycle` 调用（sideButtons、bottomButtons、advancedSettings、gestureSettings、actionSettings）
- 第 272-349 行另含约 8 个协程收集器 (`onSetOverlay` 中的 `collect` 调用)
- 总计约 13 个并行 DataStore 订阅

**建议处理**：
评估是否可以将部分设置合并为单一 DataStore 文件，或使用合并 Flow（`combine`）减少独立订阅数量。

**风险等级**：中

**是否需要用户确认**：是（涉及 DataStore 结构调整，需要确认序列化兼容性）

**是否改变用户可见行为**：否

**优先级**：P1

**推荐验证方式**：assembleDebug + 实机测试覆层响应流畅度

## 4. ViewModel / 状态管理

### 4.1 [P0] DataStoreHolder 在类加载时初始化全部 7 个 DataStore 实例

**观察事实**：
- `utils/DataStoreHolder.kt`（code_index 核心链路索引"ViewModel / 状态管理链路"的执行逻辑层）是 `object` 单例
- 第 20-60 行使用 7 个 `run { }` 块作为属性初始化器，在 `DataStoreHolder` 首次被引用时立即初始化
- 每个初始化调用 `MultiProcessDataStoreFactory.create(serializer)`，需要打开文件并创建 `Serializer` 实例
- 即使只需读取一个设置项（如 `initialSettings`），全部 7 个 DataStore 实例也被创建

**影响**：
应用启动延迟增加，尤其是在首次安装或数据清除后首次冷启动时。7 个文件同时打开，每个需要 JSON 反序列化默认值。

**建议处理**：
将 `DataStore` 属性改为 `lazy` 委托，按需延迟初始化。

**风险等级**：中

**是否需要用户确认**：否

**是否改变用户可见行为**：否（仅改变初始化时机，不影响逻辑）

**优先级**：P0

**推荐验证方式**：assembleDebug + 实机验证冷启动行为

**是否适合自动化执行**：适合自动化

### 4.2 [P1] Backup / Restore 使用 7 路并行 async 同时读写

**观察事实**：
- `utils/BackupHelper.kt`（code_index 核心链路索引"配置、资源与构建链路"的执行逻辑层）第 120-136 行 `getBackupItemBytes` 启动 7 个并行 `async`
- 第 138-181 行 `restoreBackupFromBytes` 同样启动 7 个并行 `async`
- 每个 `async` 调用 `.data.first()`（读）或 `.updateData {}`（写）

**影响**：
7 个并发文件读取和后续 7 个并发写入。在模拟存储上可能产生 I/O 争用。

**建议处理**：
限制并发数为 2-3，或使用 `awaitAll` 但确保 DataStore 内部已处理写冲突（`updateData` 是事务性的，但文件锁仍然是操作系统级）。

**风险等级**：低

**是否需要用户确认**：否

**是否改变用户可见行为**：否

**优先级**：P1

**推荐验证方式**：grep 检索 + assembleDebug

**是否适合自动化执行**：适合自动化

### 4.3 [P1] ViewModel 的 saveSettings 存在竞争条件风险

**观察事实**：
- `AdvancedSettingsVM.kt` 第 139-158 行、`HomeVM.kt` 第 203-221 行、`ActionSelectVM.kt` 第 507-617 行等在 `saveSettings()` 中使用当前 `uiState` 快照调用 `updateData`
- `HomeVM.saveSettings()` 启动 3 个并行协程，各调用 `updateData`
- 如果用户在 ViewModel 保存前快速切换多个设置，部分更改可能丢失

**影响**：
快速切换设置时，部分设置可能被覆盖而不会持久化。

**建议处理**：
将 `saveSettings` 改为使用 `combine` 或合并为单次 `updateData` 调用，或在 `updateData` 中使用 `it.copy()` 而非外部快照。

**风险等级**：中

**是否需要用户确认**：是（需确认业务期望行为——是否需要"最后生效"语义还是"每项独立保存"）

**是否改变用户可见行为**：不确定（可能修复现有竞争条件，也可能引入语义变化）

**优先级**：P1

**推荐验证方式**：grep 检索所有 `saveSettings` 调用链 + 实机快速切换设置验证

**是否适合自动化执行**：需人工确认后自动化

## 5. 无障碍服务与系统调用

### 5.1 [P1] ImeInsetObserver 创建全屏覆层窗口监听 IME

**观察事实**：
- `SideGestureService.kt`（code_index 核心链路索引"无障碍服务 / 系统调用链路"的入口文件）第 546-606 行 `ImeInsetObserver` 内部类
- 创建一个 `MATCH_PARENT × MATCH_PARENT` 的 `TYPE_ACCESSIBILITY_OVERLAY` 窗口
- 该窗口只用于监听 `onGlobalLayout` 以检测 IME 可见性
- Android 11+ 支持 `WindowInsets.Type.ime()` 无此覆层需求

**影响**：
全屏覆层窗口即使设置为 `FLAG_NOT_TOUCHABLE`，仍会增加 GPU 合成开销，并可能干扰其他无障碍服务的事件分发。

**建议处理**：
Android 11+ 上改用 `WindowInsetsCompat` 的 `ime()` 监听；低版本保留现有实现，但可按 API 版本分流。

**风险等级**：中

**是否需要用户确认**：是（涉及系统版本分流逻辑）

**是否改变用户可见行为**：否（IME 检测结果相同，只是实现方式不同）

**优先级**：P1

**推荐验证方式**：assembleDebug + 实机测试 IME 弹出/收起行为（Android 11/12/13/14）

**是否适合自动化执行**：需人工确认后自动化

### 5.2 [P1] WakeLock 超时长达 10 分钟且无早期释放

**观察事实**：
- `SideGestureServiceProxy.kt` 第 397-407 行 `KEEP_SCREEN_ON_WAKE_LOCK_TIMEOUT_MS = 10 * 60 * 1000L`（10 分钟）
- `PARTIAL_WAKE_LOCK` 仅在 toggle-off 或 `onRelease()` 时释放
- 如果 Service 在持有 WakeLock 时崩溃或被杀死，WakeLock 持续 10 分钟

**影响**：
显著电池消耗，在极端情况下可耗尽设备电量。

**建议处理**：
将超时降为更合理的值（如 1-2 分钟），或在 `onStartCommand` 返回 `START_STICKY` 的 Service 中绑定生命周期。

**风险等级**：中

**是否需要用户确认**：是（WakeLock 超时是用户可见行为的配置语义变化）

**是否改变用户可见行为**：是（"保持屏幕常亮"的保持时长变化）

**优先级**：P1

**推荐验证方式**：assembleDebug + 实机验证 WakeLock 行为

**是否适合自动化执行**：需人工确认后自动化
