## 1. 动作体系实现方式

### 1.1a [P1] actionText / actionIcon 映射数据化

**状态：已完成（commit 342edcd）**

**观察事实**：
- `ktx/GlobalActions.kt` 的 `actionText()`（第 55-110 行、第 112-168 行）和 `actionIcon()`（第 170-231 行）包含三个完整 `when` 分支，每个分支枚举相同的 30+ 动作 ID
- `actionText` 存在两个版本（`Context` 扩展函数和 `@Composable` 函数），逻辑完全相同仅获取资源方式不同（`getString` vs `stringResource`）
- 添加新动作需同步修改三个函数

**影响**：
维护负担：添加一个新动作需要同时修改 3 个位置，遗漏将导致 UI 无文本或图标。

**建议处理**：
将 actionText / actionIcon 映射替换为 `Map<String, Pair<Int, Any?>>` 数据结构，两个 `actionText` 版本共享同一个映射源。

**风险等级**：低

**是否改变用户可见行为**：否（映射关系不变，仅实现方式从 `when` 分支改为 Map 查找）

**等价性验证**：text/icon 无缺失、无新增、无错配；APP_SCREEN 保持有 text 无 icon；6 个运行时分支（EXTRA_LAUNCH_APP / EXTRA_LAUNCH_SHORTCUT × 3 函数）保持原位

**是否需要用户确认**：是（涉及函数签名和数据结构变更）

**优先级**：P1

### 1.1b [P1] onAction ActionHandler 重构

**状态：移出 v1.5.5**

**观察事实**：
- `SideGestureServiceProxy.kt`（code_index 核心链路索引"动作体系链路"）的 `onAction()` 方法包含约 250 行的 `when` 分支（对应 code_index 第 149-160 行所述"动作体系链路"的入口与执行逻辑）
- 每个分支包含不同的执行逻辑：系统 API 调用、协程、Toast、权限检查、数据反序列化、Intent 构建等
- 添加新动作需同步修改 4 个位置（`constant/GlobalActions.kt` 的 ID + `SideGestureServiceProxy.kt` 的 `onAction` + `ktx/GlobalActions.kt` 的 `actionText` + `actionIcon`）
- `onAction` 分支中的执行逻辑差异大，无法用纯数据映射替代

**影响**：
维护负担高，遗漏同步修改可导致动作 ID 有效但执行时无响应。

**建议处理**：
引入 `ActionHandler` 接口，将每个动作的执行逻辑封装到独立的实现类中，通过注册表消除 `when` 分支。

**风险等级**：中（涉及多文件重构和接口设计决策）

**是否改变用户可见行为**：否（仅内部重构，不改变动作执行结果）

**移出原因**：大架构重构，需要新建接口/实现类和包结构调整，不适合 v1.5.5

**是否需要用户确认**：是

## 2. 应用列表与应用状态查询

### 2.1 [P1] 包管理器查询存在重复循环

**状态：已审查，不做**

**风险等级**：低

**是否改变用户可见行为**：否

**不做原因**：两个函数不在同一调用链，安全复用需要改外部 API 或引入缓存

**是否需要用户确认**：否

### 2.2 [P1] findLauncherActivity 使用 Thread.sleep 阻塞

**状态：已完成（commit d2c7166）**

**观察事实**：
- `utils/AppInfoUtils.kt` 第 225-254 行 `findLauncherActivity` 使用 `Thread.sleep(retryDelayMs)` 实现重试
- 最大重试 5 次 × 200ms = 最多阻塞 1 秒
- 调用方 `SideGestureServiceProxy.kt` 已使用 `Dispatchers.IO`

**影响**：
即使运行在 IO 调度器上，阻塞仍在浪费线程资源；当应用被冻结且找不到启动 Activity 时，1 秒的重试周期完全浪费。

**建议处理**：
替换为 `delay()` 协程挂起，避免线程阻塞。

**风险等级**：低

**是否改变用户可见行为**：否

**修改内容**：findLauncherActivity 改为 suspend fun，Thread.sleep 改为 delay，CancellationException 在普通 Exception 前重抛未被吞掉

**验证**：assembleDebug + assembleRelease 通过，6 个调用方零修改

**是否需要用户确认**：否

### 2.3 [P2] Pinyin 缓存无生命周期管理

**是否需要用户确认**：否

## 3. Compose UI 与状态派生

### 3.1 [P1] 无障碍服务覆层中同时订阅 5 个 DataStore StateFlow

**状态：已审查，不做**

**观察事实**：
- `SideGestureService.kt` 第 221-270 行在 Compose 覆层中创建 5 个独立的 `collectAsStateWithLifecycle` 调用（sideButtons、bottomButtons、advancedSettings、gestureSettings、actionSettings）
- 第 272-349 行另含约 8 个协程收集器 (`onSetOverlay` 中的 `collect` 调用)
- 总计约 13 个并行 DataStore 订阅

**影响**：
每次 DataStore 文件变化触发 JSON 反序列化，所有订阅者重新计算。当多个设置项同时更新时，可在手势响应中引入可感知的延迟。

**建议处理**：
评估是否可以将部分设置合并为单一 DataStore 文件，或使用合并 Flow（`combine`）减少独立订阅数量。

**风险等级**：中

**是否改变用户可见行为**：否

**不做原因**：
1. 订阅生命周期清晰，Composable 层通过 `Lifecycle.repeatOnLifecycle`，coroutine 层在 `onDestroy` 统一 `cancel`，无泄漏
2. DataStore Flow 扇出开销极低（微秒级 JSON 反序列化 + 5 个 lambda 调用）
3. DataStore 变更发生在用户交互时期，不在手势执行路径上
4. 合并订阅属于架构重构，收益低

**是否需要用户确认**：是

## 4. ViewModel / 状态管理

### 4.1 [P0] DataStoreHolder 在类加载时初始化全部 7 个 DataStore 实例

**风险等级**：中

**是否改变用户可见行为**：否（仅改变初始化时机，不影响逻辑）

**验证**：assembleDebug + assembleRelease 通过，实机回归通过

**是否需要用户确认**：否

### 4.2 [P1] Backup / Restore 使用 7 路并行 async 同时读写

**状态：已审查，不做**

**观察事实**：
- `utils/BackupHelper.kt`（code_index 核心链路索引"配置、资源与构建链路"的执行逻辑层）第 120-136 行 `getBackupItemBytes` 启动 7 个并行 `async`
- 第 138-181 行 `restoreBackupFromBytes` 同样启动 7 个并行 `async`
- 每个 `async` 调用 `.data.first()`（读）或 `.updateData {}`（写）

**影响**：
7 个并发文件读取和后续 7 个并发写入。在模拟存储上可能产生 I/O 争用。

**建议处理**：
限制并发数为 2-3，或使用 `awaitAll` 但确保 DataStore 内部已处理写冲突（`updateData` 是事务性的，但文件锁仍然是操作系统级）。

**风险等级**：低

**是否改变用户可见行为**：否

**不做原因**：
1. 备份路径实际是 `async { }.await()` 的**顺序执行**（Kotlin 构造参数从左到右求值），不存在 7 路并行
2. 恢复路径 7 个 `updateData` 写入 7 个不同 DataStore 文件，无文件锁竞争
3. 每个文件仅数百字节，I/O 量极小

**是否需要用户确认**：否

### 4.3 [P1] ViewModel 的 saveSettings 存在竞争条件风险

**状态：已审查，不做**

**观察事实**：
- `AdvancedSettingsVM.kt` 第 139-158 行、`HomeVM.kt` 第 203-221 行、`ActionSelectVM.kt` 第 507-617 行等在 `saveSettings()` 中使用当前 `uiState` 快照调用 `updateData`
- `HomeVM.saveSettings()` 启动 3 个并行协程，各调用 `updateData`
- 如果用户在 ViewModel 保存前快速切换多个设置，部分更改可能丢失

**影响**：
快速切换设置时，部分设置可能被覆盖而不会持久化。

**建议处理**：
将 `saveSettings` 改为使用 `combine` 或合并为单次 `updateData` 调用，或在 `updateData` 中使用 `it.copy()` 而非外部快照。

**风险等级**：中

**是否改变用户可见行为**：不确定（可能修复现有竞争条件，也可能引入语义变化）

**不做原因**：
1. 当前不是明确真实竞态——`Dispatchers.Main` 单线程 + lambda 执行时读取 live `uiState` 而非闭包快照 + 无外部竞争者
2. 正确修复需重构 setter/save 语义（传具体变更而非 dump 全量列表），最小改动可能改变"最后生效"语义
3. 若只做最小改动（uiState.xxx → it），saveSettings 不知道哪些按钮变了，无法在 `it` 基础上正确合并

**是否需要用户确认**：是

### 5.3 [P2] GestureResultCallback 匿名对象在每次手势分派时创建

## 6. 工具类与通用能力

### 6.1 [P0] JsonHelper.encodeDefaults = true 导致所有序列化输出包含默认值

**状态：已完成（commit 456bf7e）**

**观察事实**：
- `utils/JsonHelper.kt`（code_index 第 20-60 行统计中"工具类"分类）第 13 行 `encodeDefaults = true`
- 所有 DataStore JSON 文件默认值字段被序列化，如 `GestureButton.alignRegion = true`、`AdvancedSettings` 的约 15 个布尔/枚举字段的默认值
- `GestureButton` 有 12 个字段，`AdvancedSettings` 约 15 个字段，序列化时全部显式写入

**影响**：
每个 DataStore 文件体积膨胀约 30-50%，读写速度相应降低。每次启动时反序列化额外数据增加冷启动时间。

**建议处理**：
移除 `encodeDefaults = true`。`kotlinx.serialization` 在不设置此选项时，仅序列化非默认值的字段。DataStore 的 `readFrom` 使用 `ignoreUnknownKeys = true` 已确保向后兼容。

**风险等级**：低

**是否改变用户可见行为**：否

**兼容性核验结论**：
- 旧 JSON → 新 reader：兼容（`ignoreUnknownKeys = true`）
- 新 JSON → 旧 reader：兼容（缺失字段从构造器默认值补齐）
- Backup 兼容：兼容（Backup 全 nullable 字段）
- DataStore 首次读取：兼容（`defaultValue` = 构造器默认值）

**是否需要用户确认**：否

### 6.2 [P1] Toast 使用 Legacy Handler 而非协程

**状态：移出 v1.5.5**

**观察事实**：
- `utils/Toast.kt`（code_index 核心链路索引中多次引用的工具类）第 20、58、67、77、86 行使用 `Handler(Looper.getMainLooper())` + `postDelayed`
- 调用方已有 `viewModelScope`（VM 中）或 `coroutineScope`（Service 中）

**影响**：
`Handler` 回调即使在 ViewModel 清除或 Service 销毁后仍会执行，可能导致在已销毁的引用上调用 `continueBlock()`。

**建议处理**：
替换为 `viewModelScope.launch { delay(); block() }` 或 `serviceScope.launch { delay(); block() }`。

**风险等级**：低

**是否改变用户可见行为**：否

**移出原因**：
1. Handler 仅用于延迟 `continueBlock`，是合理 Android 用法
2. 底层 `showComposeToast` 已使用 `GlobalScope.launch` + Channel
3. 改为协程需要改所有 20+ 调用方或添加 CoroutineScope 参数

**是否需要用户确认**：否
