## 1. 动作体系实现方式


### 1.1b [P1] onAction ActionHandler 重构

**状态：移出 v1.5.6**

**观察事实**：
- `SideGestureServiceProxy.kt`（code_index 核心链路索引"动作体系链路"）的 `onAction()` 方法包含约 250 行的 `when` 分支（对应 code_index 第 149-160 行所述"动作体系链路"的入口与执行逻辑）
- 每个分支包含不同的执行逻辑：系统 API 调用、协程、Toast、权限检查、数据反序列化、Intent 构建等
- 添加新动作需同步修改 4 个位置（`constant/GlobalActions.kt` 的 ID + `SideGestureServiceProxy.kt` 的 `onAction` + `ktx/GlobalActions.kt` 的 `actionText` + `actionIcon`）
- `onAction` 分支中的执行逻辑差异大，无法用纯数据映射替代

**影响**：
维护负担高，遗漏同步修改可导致动作 ID 有效但执行时无响应。

**建议处理**：
引入 `ActionHandler` 接口，将每个动作的执行逻辑封装到独立的实现类中，通过注册表消除 `when` 分支。

**风险等级**：中（涉及多文件重构和接口设计决策）

**是否改变用户可见行为**：否（仅内部重构，不改变动作执行结果）

**移出原因**：大架构重构，需要新建接口/实现类和包结构调整，不适合 v1.5.6

**是否需要用户确认**：是

### 1.2 [P1] 遗留 Legacy 动作 ID 的完整分支仍存活

**状态：已审查，保留不动**

**观察事实**：

**建议处理**：
确认从 DataStore 获取的旧配置是否仍可能包含这些 ID。如果旧配置升级路径已完成，可删除对应分支。

**风险等级**：低

**是否改变用户可见行为**：否（仅删除不可达分支）

**不做原因**：
1. 342edcd 后 legacy ID 是 Map 条目和常量，不再是 when-branch
2. 直接删除会让旧配置 UI 降级（从显示旧名称变为显示"None"）
3. 无现有 migration 入口，清理需设计 legacy action normalization 方案

**是否需要用户确认**：是

### 1.3 [P2] Action 数据以原始 String 传递，无编译期类型安全

## 2. 应用列表与应用状态查询

### 2.1 [P1] 包管理器查询存在重复循环

**状态：已审查，不做**

**观察事实**：
- `utils/AppInfoUtils.kt`（code_index 核心链路索引"应用列表链路"的执行逻辑层）中 `queryFrozenApplications` 和 `inspectDisabledAppsByPackageManager` 各自独立调用 `pm.getInstalledApplications(0)` 并遍历

**影响**：

**建议处理**：
将 `pm.getInstalledApplications(0)` 结果缓存或合并为一次遍历。

**风险等级**：低

**是否改变用户可见行为**：否

**不做原因**：两个函数不在同一调用链，安全复用需要改外部 API 或引入缓存

**是否需要用户确认**：否

### 2.3 [P2] Pinyin 缓存无生命周期管理

**状态：已审查，不做**

**观察事实**：
- `utils/AppSearchUtils.kt` 第 11-15 行 `pinyinIndexCache` 是顶层 `LinkedHashMap`，最大 1024 条

**是否需要用户确认**：否

## 3. Compose UI 与状态派生

### 3.1 [P1] 无障碍服务覆层中同时订阅 5 个 DataStore StateFlow

**状态：已审查，不做**

**观察事实**：
- `SideGestureService.kt` 第 221-270 行在 Compose 覆层中创建 5 个独立的 `collectAsStateWithLifecycle` 调用（sideButtons、bottomButtons、advancedSettings、gestureSettings、actionSettings）
- 第 272-349 行另含约 8 个协程收集器 (`onSetOverlay` 中的 `collect` 调用)
- 总计约 13 个并行 DataStore 订阅

**影响**：
每次 DataStore 文件变化触发 JSON 反序列化，所有订阅者重新计算。当多个设置项同时更新时，可在手势响应中引入可感知的延迟。

**建议处理**：
评估是否可以将部分设置合并为单一 DataStore 文件，或使用合并 Flow（`combine`）减少独立订阅数量。

**风险等级**：中

**是否改变用户可见行为**：否

**不做原因**：
1. 订阅生命周期清晰，Composable 层通过 `Lifecycle.repeatOnLifecycle`，coroutine 层在 `onDestroy` 统一 `cancel`，无泄漏
2. DataStore Flow 扇出开销极低（微秒级 JSON 反序列化 + 5 个 lambda 调用）
3. DataStore 变更发生在用户交互时期，不在手势执行路径上
4. 合并订阅属于架构重构，收益低

**是否需要用户确认**：是

### 3.4 [P2] ActionSelectScreen 过滤器链多次遍历

## 4. ViewModel / 状态管理


### 4.2 [P1] Backup / Restore 使用 7 路并行 async 同时读写

**状态：已审查，不做**

**观察事实**：
- `utils/BackupHelper.kt`（code_index 核心链路索引"配置、资源与构建链路"的执行逻辑层）第 120-136 行 `getBackupItemBytes` 启动 7 个并行 `async`
- 第 138-181 行 `restoreBackupFromBytes` 同样启动 7 个并行 `async`
- 每个 `async` 调用 `.data.first()`（读）或 `.updateData {}`（写）

**影响**：
7 个并发文件读取和后续 7 个并发写入。在模拟存储上可能产生 I/O 争用。

**建议处理**：
限制并发数为 2-3，或使用 `awaitAll` 但确保 DataStore 内部已处理写冲突（`updateData` 是事务性的，但文件锁仍然是操作系统级）。

**风险等级**：低

**是否改变用户可见行为**：否

**不做原因**：
1. 备份路径实际是 `async { }.await()` 的**顺序执行**（Kotlin 构造参数从左到右求值），不存在 7 路并行
2. 恢复路径 7 个 `updateData` 写入 7 个不同 DataStore 文件，无文件锁竞争
3. 每个文件仅数百字节，I/O 量极小

**是否需要用户确认**：否

### 4.3 [P1] ViewModel 的 saveSettings 存在竞争条件风险

**状态：已审查，不做**

**观察事实**：
- `AdvancedSettingsVM.kt` 第 139-158 行、`HomeVM.kt` 第 203-221 行、`ActionSelectVM.kt` 第 507-617 行等在 `saveSettings()` 中使用当前 `uiState` 快照调用 `updateData`
- `HomeVM.saveSettings()` 启动 3 个并行协程，各调用 `updateData`
- 如果用户在 ViewModel 保存前快速切换多个设置，部分更改可能丢失

**影响**：
快速切换设置时，部分设置可能被覆盖而不会持久化。

**建议处理**：
将 `saveSettings` 改为使用 `combine` 或合并为单次 `updateData` 调用，或在 `updateData` 中使用 `it.copy()` 而非外部快照。

**风险等级**：中

**是否改变用户可见行为**：不确定（可能修复现有竞争条件，也可能引入语义变化）

**不做原因**：
1. 当前不是明确真实竞态——`Dispatchers.Main` 单线程 + lambda 执行时读取 live `uiState` 而非闭包快照 + 无外部竞争者
2. 正确修复需重构 setter/save 语义（传具体变更而非 dump 全量列表），最小改动可能改变"最后生效"语义
3. 若只做最小改动（uiState.xxx → it），saveSettings 不知道哪些按钮变了，无法在 `it` 基础上正确合并

**是否需要用户确认**：是

### 4.4 [P1] Events 使用 GlobalScope 发起协程

**状态：移出 v1.5.6**

**观察事实**：
- `utils/Events.kt`（code_index"工具类与通用能力链路"的执行逻辑层）第 18-27 行，当 `post()` 从非主线程调用且 `postOnUiThread=true` 时，使用 `GlobalScope.launch(Dispatchers.Main)`

**影响**：

**风险等级**：低

**是否改变用户可见行为**：否

**移出原因**：问题成立但收益低。`GlobalScope` → `MainScope()` 仅消除 `@DelicateCoroutinesApi` 警告；传 scope 参数会改变 `Events.post` API

**是否需要用户确认**：否

**风险等级**：中

**是否改变用户可见行为**：否（IME 检测结果相同，只是实现方式不同）

**是否需要用户确认**：是（涉及系统版本分流逻辑）

### 7.2 [P1] MultiProcessDataStoreFactory 在单进程应用中不必要

**状态：移出 v1.5.6**

**观察事实**：
- `ktx/DataStore.kt` 第 40 行使用 `MultiProcessDataStoreFactory.create`
- SideLeap 的无障碍服务（`SideGestureService`）和主 Activity 运行在同一进程（查看 AndroidManifest 确认）
- Shizuku BridgeService 同样运行在应用进程内

**影响**：
`MultiProcessDataStoreFactory` 添加文件级锁开销（操作系统 `flock`），每个读写操作增加额外同步延迟。

**建议处理**：
替换为 `DataStoreFactory.create`（单进程工厂）。如果未来需要多进程访问，再切换回 `MultiProcessDataStoreFactory`。

**风险等级**：低

**是否改变用户可见行为**：否

**移出原因**：涉及 DataStore 结构调整，需确认进程模型

**是否需要用户确认**：是（需确认所有 DataStore 访问确实在单进程内）

## 8. 构建、资源与发布链路

### 8.1 [P1] Debug 构建包含无意义的 ProGuard 配置

**状态：移出 v1.5.6**

**观察事实**：
- `app/build.gradle.kts`（code_index"构建、资源与发布链路"的入口文件）第 50-57 行，`debug` buildType 虽 `isMinifyEnabled = false`，但仍指定 `proguardFiles(...)`
- `perf` buildType（第 58-66 行）使用 `initWith(getByName("release"))` 但 `signingConfig = signingConfigs.getByName("debug")`

**影响**：
Gradle 在 debug 构建时仍解析 ProGuard 规则文件，增加 ~2-3 秒构建时间。`perf` buildType 引用的 `debug` signingConfig 可能不存在。

**建议处理**：
从 `debug` buildType 移除 `proguardFiles`。确认 `perf` buildType 的 `signingConfig` 引用正确。

**风险等级**：低

**是否改变用户可见行为**：否

**移出原因**：涉及构建配置变更

**是否需要用户确认**：是

### 8.2 [P1] ProGuard 规则缺少 Compose / kotlinx.serialization 关键规则

**状态：移出 v1.5.6**

**观察事实**：
- `app/proguard-rules.pro`（第 23-26 行）仅保留 entity 包和 Shizuku 服务类
- `-keep class hunoia.sideleap.entity.** { *; }` 过于宽泛，阻止了 entity 类的优化和混淆
- 缺少 `@Composable` 函数、`@Stable` 注解、`kotlinx.serialization` 序列化器、`kotlinx.coroutines` 的 Keep 规则

**影响**：
如果未来在 release 构建中启用更激进的 R8 优化，Compose 运行时可能因混淆而中断。当前未出现问题是因为 `isMinifyEnabled = true` 但 R8 未完全优化 Compose 路径。

**风险等级**：高（潜在——开启全量优化后可能出问题而未及时发现）

**是否改变用户可见行为**：不确定（取决于是否同时启用 full R8 mode）

**移出原因**：高风险，需要全面 Release 验证

**是否需要用户确认**：是
