# SideLeap 分支管理规范

本文档定义 SideLeap 的长期分支管理规则。除非项目规模、协作模式或发布流程发生明显变化，后续开发、修复、架构整理和发布均遵循本规范。

---

## 基本原则

SideLeap 采用：

```text
main + 短生命周期任务分支
```

核心原则：

- `main` 始终保持稳定、可构建、可发布。
- 所有新功能、架构整理、bug 修复都从 `main` 新建独立分支。
- 一个分支只解决一个明确目标。
- 一个 commit 只表达一个清晰修改目的。
- 分支完成后及时合并或废弃，不长期悬挂。
- 不在不稳定分支上继续叠加新功能或新架构阶段。
- 合并前必须完成构建验证，并列出需要人工确认的测试内容。
- 发现方向错误时优先回滚或重开分支，不在坏分支上连续补丁。

---

## 分支类型

| 类型 | 用途 |
|---|---|
| `feat/*` | 新功能分支 |
| `refactor/*` | 架构整理分支 |
| `fix/*` | 修复分支 |
| `docs/*` | 文档分支 |
| `chore/*` | 维护分支 |

---

## 分支创建规则

开始任务前，必须先回到最新 `main`。

确认工作区干净后创建分支。

不从旧功能分支继续派生新功能分支，除非该任务明确依赖前一个未合并分支。

---

## 合并规范

- 所有任务分支合并到 `main` 必须使用 `git merge --no-ff`，禁止 fast-forward 合并。
- 这样保证每次合并留下显式的 merge commit，git 图上可追溯分叉与汇合记录。
- 合并前必须完成构建验证和人工测试确认。

## 提交规范

commit 格式：

